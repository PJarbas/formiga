## Codebase Patterns
- Control-server endpoints in `src/server/control-server.ts` handle pause/resume via HTTP POST to `/control/pause-run` and `/control/resume-run`, authenticated with `x-tamandua-secret`
- Events are emitted via `emitEvent()` from `src/installer/events.ts`, which writes to both run-specific (`events/<runId>.jsonl`) and global (`events/all.jsonl`) JSONL files
- Tests for control-server spawn the daemon as a child process with temp HOME, then exercise endpoints over HTTP
- Control-server tests insert runs directly into SQLite via `node:sqlite` DatabaseSync
- `handleResumeRun` emits the event BEFORE calling `handleRegisterRun`, so events are emitted even if registration subsequently fails
- `finalizeDrainingPause()` checks scheduling_status and running step count; only acts when status is draining_pause AND no running steps remain; safe to call from any step transition point
- When calling `finalizeDrainingPause` from tests, must set both `TAMANDUA_DB_PATH` and `TAMANDUA_STATE_DIR` so `getDb()` and `emitEvent()` target the same state as the daemon
- `executePollingRound` already checks run status before spawning work; the `draining_pause` guard was already in place at line ~1060 of `agent-scheduler.ts`
- CLI tests that exercise daemon-related commands must set `TAMANDUA_CONTROL_PORT` to an unused port to avoid accidentally hitting a running control plane; otherwise `pauseRunWithDaemon` returns a 404 response (not null), making daemon-unreachable assertions fail
- CLI action handlers that need a target (like `pause`, `stop`, `resume`) must be placed BEFORE the `if (!target) { printUsage(); process.exit(1); }` guard in `cli.ts`, otherwise they're never reached when target is missing
- `resumeRunWithDaemon` can return status 200 or 202 (from `admitOrQueueRun`); the CLI handler must accept both as success
- `resumeWorkflow` (for failed runs) calls `ensureDaemonControlAvailable()` and `registerRunWithDaemon`, requiring a running daemon; CLI tests for this path need a daemon spawn
- The real `steps` table schema uses `id TEXT PRIMARY KEY` (separate from `step_id`); tests inserting steps must match this schema
- CLI resume tests with daemon MUST copy the workflow directory into the temp home so `admitOrQueueRun` can load the workflow spec via `loadWorkflowSpec(resolveWorkflowDir(run.workflow_id))`
- `pauseRunWithDaemon` in `control-client.ts` now accepts an optional `drain` boolean parameter (default false); when true, passes `{ runId, drain: true }` in the request body
- Dashboard UI tests use `startDashboard()`/`stopDashboard()` helper pattern, fetching `/` and checking HTML content with `assert.match()` against the full HTML (not just JS source)
- The runs table header HTML is embedded in the `renderRuns()` JS function as a template literal; tests matching table structure must check the full HTML output, not just the separate JS section
- When adding columns to the dashboard runs table, any test with a regex matching the `<th>` header sequence must be updated (e.g., `tests/dashboard-run-token-spend.test.ts`)
- MCP server uses `createProtocolServer` to wire tools via `CallToolRequestSchema`; tools are defined in `mcpTools` array with full JSON Schema input/output
- MCP tests use `startTamanduaMcpServer(0, { services })` for DI-based testing; mock services return lightweight data and capture call arguments for verification
