## Codebase Patterns
- Tests import from `../../dist/...` (compiled JS), not from source TypeScript directly
- agent-scheduler.ts uses module-private jobMetadata Map; test helpers follow `_functionName` convention for exposing internals
- setupAgentCrons() intervalMinutes is computed inline when calling createAgentCronJob() — Math.max(floor, ceil(timeoutSeconds/60)) when polling.timeoutSeconds is set, fallback default otherwise
- Agent-scheduler exports are re-exported through agent-cron.ts barrel for backward compat
- Tests run with `node --test src/**/*.test.ts` — TypeScript test files are excluded from tsc build but run directly via Node's type stripping
- `SetupAgentCronsOptions` is the options bag for setupAgentCrons(); it controls workingDirectoryForHarness and now noHurrySaveTokensMode
- runWorkflow() seeds context into the DB runs.context JSON column BEFORE daemon registration, so tests can catch the expected daemon error and still read the DB to verify context
- For DB queries in tests, lazily import getDb: `const { getDb } = await import("../../dist/db.js");` — avoids module-load ordering issues with TAMANDUA_DB_PATH
- `_admitOrQueueRun` test export convention in control-server.ts follows agent-scheduler's `_functionName` pattern for exposing internals
- control-server integration tests that need a real workflow must create a minimal workflow.yml in `<stateDir>/workflows/<id>/` on disk; `loadWorkflowSpec` validates agents have id + workspace.baseDir, and steps have id + agent
- `shutdownAllCrons()` must be called in afterEach to clean up in-memory jobMetadata between tests that use `_admitOrQueueRun`
