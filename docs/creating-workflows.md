# Creating Workflows

This guide walks through creating a custom workflow for Tamandua.

## Overview

A workflow is a directory containing:
- `workflow.yml` — the workflow specification
- `agents/<agent-id>/` — each agent's persona files (AGENTS.md, IDENTITY.md, SOUL.md)

## workflow.yml

```yaml
id: my-workflow           # Unique identifier (kebab-case)
name: My Custom Workflow   # Human-readable name
version: 1                  # Schema version

# Agents: each runs a specific role in the pipeline
agents:
  - id: planner
    name: Planner
    role: analysis           # Optional: analysis|coding|verification|testing|pr|scanning
    model: claude-sonnet     # Optional: model override
    timeoutSeconds: 1800     # Optional: time limit (default: 20-30 min by role)
    workspace:
      files:
        AGENTS.md: agents/planner/AGENTS.md
        IDENTITY.md: agents/planner/IDENTITY.md
        SOUL.md: agents/planner/SOUL.md
      skills:                # Optional: skill directories to install
        - my-custom-skill

  - id: developer
    name: Developer
    role: coding
    timeoutSeconds: 3600
    workspace:
      files:
        AGENTS.md: agents/developer/AGENTS.md
        IDENTITY.md: agents/developer/IDENTITY.md
        SOUL.md: agents/developer/SOUL.md

# Steps: ordered pipeline — each step is executed by one agent
steps:
  - id: plan
    agent: planner
    input: |
      Plan the implementation of {{task}}. 
      Break it into stories and output them as STORIES_JSON.
      Reply with STATUS: done when complete.
    expects: "STATUS: done"

  - id: implement
    agent: developer
    type: loop              # Loop over stories
    loop:
      over: stories
      completion: all_done
      verifyEach: true
      verifyStep: verify
    input: |
      Implement {{story.title}}:
      {{story.description}}
      Acceptance Criteria:
      {{story.acceptance_criteria}}
      Reply with STATUS: done and CHANGES: what you changed.
    expects: "STATUS: done"
    max_retries: 2
    on_fail:
      retry_step: implement
      max_retries: 3
      on_exhausted:
        escalate_to: user

  - id: verify
    agent: verifier
    input: |
      Verify {{story.title}} against acceptance criteria.
      Run tests and confirm everything works.
      Reply with STATUS: done or STATUS: fail.
    expects: "STATUS: done"
```

## Agent Persona Files

Each agent needs at minimum an `AGENTS.md` file in its workspace:

```markdown
# AGENTS.md

You are a workflow agent in the Tamandua system.
Your role: Planner
You decompose tasks into implementable stories.

Always format your output with KEY: value lines:
STATUS: done
PLAN: your plan summary

For stories, output them as STORIES_JSON: [{"id": "S1", "title": "...", ...}]
```

Optionally add:
- `SOUL.md` — personality and behavior guidelines
- `IDENTITY.md` — agent identity and background

## Step Input Templates

Step inputs use `{{key}}` placeholders that are resolved at runtime:

| Placeholder | Description |
|-------------|-------------|
| `{{task}}` | The task description from `tamandua workflow run` |
| `{{context.key}}` | Any key from the workflow's context |
| `{{story.id}}` | Story ID (loop steps only) |
| `{{story.title}}` | Story title (loop steps only) |
| `{{story.description}}` | Story description (loop steps only) |
| `{{story.acceptance_criteria}}` | Formatted acceptance criteria |

Context from previous steps is also available — each step's KEY: value output is collected and made available as `{{key}}` in subsequent steps.

## Step Output Format

Agents must format their output with specific KEY: value lines:

```
STATUS: done
CHANGES: what was changed
TESTS: what tests were run
```

For loop steps that produce stories:
```
STORIES_JSON: [{"id": "S1", "title": "Add login", "description": "...", "acceptanceCriteria": ["Users can log in", "..."]}]
STATUS: done
PLAN: implementation plan summary
```

## Roles

| Role | Capabilities | Use For |
|------|-------------|---------|
| `analysis` | Read code, run git/grep, reason | Planner, reviewer, investigator, triager |
| `coding` | Full read/write/exec | Developer, fixer, setup |
| `verification` | Read + exec, no write | Verifier |
| `testing` | Read + exec + browser/web | Tester |
| `pr` | Read + exec only | PR creation |
| `scanning` | Read + exec + web search | Security scanner |

## Retry and Escalation

```yaml
on_fail:
  retry_step: implement    # Step to retry
  max_retries: 3            # Max retries before escalation
  on_exhausted:
    escalate_to: user       # "user" or another step id
```

## Loops

Loop steps repeat over stories generated by a prior step:

```yaml
- id: implement
  type: loop
  loop:
    over: stories           # Iterate over stories
    completion: all_done    # Done when all stories completed
    verifyEach: true        # Run verify step after each story
    verifyStep: verify      # Which step does verification
    freshSession: true      # New session per story
```

## Installing Custom Workflows

```bash
# During development, point to a local workflow
tamandua workflow install /path/to/my-workflow

# Or bundle it in your own repo
tamandua workflow install github.com/my-org/my-workflows --workflow my-workflow
```
