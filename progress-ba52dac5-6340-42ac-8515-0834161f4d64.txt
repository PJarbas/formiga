## Codebase Patterns

- daemonctl.ts uses `os.homedir()` at module load for all file path constants (MCP_PID_FILE, MCP_PORT_FILE, etc.)
- startMcp(), startDaemon(), startControlPlane() all accept `{ homeDir }` option for test isolation
- stopMcp(), stopDaemon(), stopControlPlane() do NOT accept homeDir — they always use module-level constants resolved against real HOME
- Tests import from dist/server/daemonctl.js (compiled output), not source
- Tests use node:test with describe/it/before/after
- canBind() pattern: create server, try listening, close — returns true/false
- reserveRandomPort() pattern: create server, listen(0), capture port, close — avoids hardcoded ports
- Isolated test pattern: createTempHome (fs.mkdtempSync), use startMcp/startControlPlane(port, { homeDir: tempHome }), write isolated helpers that mirror the real API but use path.join(tempHome, '.tamandua', ...) instead of module-level constants, cleanup with fs.rmSync(tempHome, { recursive: true, force: true }) in finally
- CLI integration test pattern (mcp-cli.test.ts, control-plane-cli.test.ts): create temp HOME, pass HOME env to CLI subprocess, use isolated PID/port file helpers mirroring daemonctl API, cleanup both real HOME (belt) and isolated HOME in before/after/finally
- Control plane isolation mirrors MCP isolation pattern: same helper shapes (createXxxTempHome, getIsolatedXxxPidFile, getIsolatedXxxPortFile, readIsolatedXxxPort, writeIsolatedXxxPort, isIsolatedXxxRunning, stopIsolatedXxx, cleanupIsolatedXxxFiles)
- Daemon spawning with isolated control plane: `getControlPort()` in daemon.ts reads `TAMANDUA_CONTROL_PORT` env var (defaults to 3339). To spawn a daemon with zero port contention, pass `TAMANDUA_CONTROL_PORT` env var with a random reserved port alongside `HOME` for temp home isolation.
- Default port assertion pattern: when a test checks negative reachability on a default port (e.g., MCP on 3338), use `canBind()` pre-check. If the default port is occupied by a real daemon, skip the assertion rather than failing — the test can't disambiguate between "daemon started a service" and "something else is on the port."
- DEFAULT_MCP_PORT binding pattern for dashboard-status-mcp.test.ts: tests that start MCP on DEFAULT_MCP_PORT (3338) use `t.skip()` (not `assert.fail()`) when `canBind(DEFAULT_MCP_PORT)` returns false — another test or real daemon may be using it. PID/port file isolation is achieved via temp HOME (createTempEnv → cliEnv with HOME), not custom ports. Tests that conditionally start MCP use `if (await canBind(...)) { ... }` without failing when the port is occupied.
