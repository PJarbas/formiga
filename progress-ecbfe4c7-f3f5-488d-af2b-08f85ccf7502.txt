## Codebase Patterns
- Dashboard routes are handled in `src/server/dashboard.ts` `route()` function
- Route registration order matters: more specific routes (e.g. `/api/runs/:id/autoresearch`) must be registered before generic ones (e.g. `/api/runs/:id`)
- `resolveRunHarnessCwd()` resolves harness cwd from run context (checking `working_directory_for_harness`, `worktree_path`, `cwd`)
- `findAutoresearchSessionCwd()` walks directory tree looking for `autoresearch.config.json`
- Tests use `createDashboardServer(0)` for ephemeral servers, set `TAMANDUA_DB_PATH` to use temp DBs
- DB runs are inserted directly via `getDb().prepare().run()` in tests
- Dashboard tests use same shape as `/api/runs` response: `{ runs: Array<{ id, workflow_id, task, status, ... }> }`
- CLI command renames follow a consistent pattern: update the help function name, update all help text strings, update dispatch in both --help block and action switch, update error messages, update template references, update tests
- CLI tests using `autoresearch init` for loop tests must pass `--cwd` to init as well, otherwise the session is created in repo root not tmpDir
- When testing autoresearch loop via CLI, use `makeTestEnv()` for a shared tmpDir across init+loop since `cli()` creates a new testEnv per call
- When renaming commands, old command names must produce "Unknown autoresearch action: <name>" with exit code 1 and show valid subcommands in error message (without the old name)
- `dist/cli/cli.js` must be rebuilt after source edits before testing; the globally installed `tamandua` symlink may point to a different checkout
- Pre-existing flaky test failures in MCP/daemon lifecycle tests are normal; they depend on process timing and vary across runs. Always re-run to confirm regressions vs. flaky.
