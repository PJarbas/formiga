## Codebase Patterns

- `kanban-data.ts` exports `buildKanbanSnapshot`, `buildKanbanCardDetail`, and related types for the kanban view
- Tests import from `../../dist/server/` (compiled output), must build before running tests
- Dashboard routes in `dashboard.ts` use regex matching on pathname, with more specific routes registered before less specific ones
- Events are `TamanduaEvent` objects from `src/installer/events.ts` — read via `getRunEvents(runId)`
- Steps table has `input_template` (the prompt), `output`, `type` (single/loop), `step_id`, `agent_id`
- Stories table has `story_id`, `title`, `description`, `acceptance_criteria` (JSON string), `output`, `status`
- Event types: `story.started`, `story.done`, `story.failed`, `step.running`, `step.done`, `step.failed`, `run.tokens.updated`
- SQLite columns use snake_case; TypeScript interfaces use camelCase and cast via `as unknown as`
- Tests use `DatabaseSync(":memory:")` for isolated in-memory SQLite
- `buildKanbanCardDetail` accepts optional `runEvents` parameter for test injection; dashboard passes `getRunEvents(runId)`
- Event filtering for story cards: include events where storyId matches OR (stepId matches loop step AND storyId is absent)
- `safeParseJsonArray` returns undefined for empty arrays to distinguish from truly absent data
- Kanban HTML renderBoard card template: id-row → title → meta-row; toggle button goes between id-row and title
- Kanban card classes: `.card`, `.card.running`, `.card.done`, `.card.failed` — status color via border-left
- Dashboard tests for kanban parse the HTML body of the kanban view served by the server; CSS class presence verifiable via regex/includes
- `setAttribute("data-card-id", ...)` sets data attribute on card element at runtime (not visible in raw server HTML)
- Test assertions for JS code presence in HTML use `.includes()` for function names and patterns
