## Codebase Patterns
- WorkflowSpec type lives in `src/installer/types.ts`
- Workflow parsing/validation lives in `src/installer/workflow-spec.ts`
- Workflow validation tests are in `tests/workflow-validation.test.ts` (import from `dist/`)
- Unit tests for installer modules live alongside source: `src/installer/*.test.ts`
- Tests use `node:test` and `node:assert/strict`
- Build: `npm run build` (tsc + html copy + version inject)
- Test: `node --test 'src/**/*.test.ts' 'tests/*.test.ts'`
- YAML parsing uses the `yaml` package (`parse as parseYaml`)
- DB migration pattern: use `CREATE TABLE IF NOT EXISTS` for idempotency, `CREATE INDEX IF NOT EXISTS` for indexes
- DB tests use isolated temp HOME directories to avoid touching the real `~/.tamandua/tamandua.db`
- Column discovery: `PRAGMA table_info(<table>)` returns `{ name, type, notnull, pk }` rows
- Table existence check: `sqlite_master WHERE type='table' AND name=?`
- DB module caches connection with 5s TTL and re-migrates on each `getDb()` call, so migrations must be idempotent
- `git rev-parse --git-common-dir` returns relative paths from origin repo (via `-C`) but absolute paths from inside a worktree — always normalize with `path.isAbsolute()`/`path.resolve()`
- `git worktree remove` must be run from the origin repo (`-C <origin>`), not from the worktree itself
- `git worktree add --detach` creates a worktree with HEAD at the target ref but no branch checked out
- `git branch --show-current` returns empty string in detached HEAD state
- `run_worktrees.worktree_origin_ref` and `worktree_origin_sha` columns are nullable in the actual migration (US-002) — row-to-type mapping needs null coalescence (`?? ""`)
- Git commands in installer modules use `spawnSync` with argument arrays (not shell strings), matching `run-harness.ts` pattern
- CLI integration tests that run `workflow run` must use `runCliUntilOutput` (not `runCliToExit`) because workflow run spawns background daemon/polling processes that don't exit
- CLI subcommands follow the pattern: `group` = args[0], `action` = args[1], `target` = args[2]
- New top-level command groups are added before the final `if (group !== "workflow")` guard
- `validateRunHarnessForScheduling` checks workspace_mode first for worktree runs, delegating to `validateRunWorktree` before generic workdir checks — this ensures worktree-specific error messages surface correctly
- RESERVED_CONTEXT_KEYS protects structural context keys from being overwritten by step output parsing; add any new harness-level context keys here
- MCP server services pattern: injectable via `TamanduaMcpToolServices`; each tool handler uses services for logic; tests mock services to avoid filesystem/DB dependencies
- MCP tools declare `inputSchema` with `required`, `properties`, `additionalProperties: false`; tool descriptions in `mcpTools` array
- `resolveWorkflowDir` + `loadWorkflowSpec` are injected as a service (`resolveWorkspaceMode`) so MCP tests can mock workspace mode without real workflow directories
- Worktree enrichment pattern: `status.ts` `buildRunDetail()` parses `runs.context` JSON to check `workspace_mode`, then queries `run_worktrees` table; same pattern used in `dashboard.ts` `handleRunDetail`
- `RunDetail` includes optional worktree fields (`workspace_mode`, `worktree_path`, `worktree_origin_repository`, `worktree_origin_ref`, `worktree_origin_sha`) — populated only for worktree runs
- `RunInfo` (compact list) does NOT include any worktree fields — keeps `workflow runs` output unchanged
- Dashboard API run detail response adds a `worktree` key (object or null) containing worktree fields from `run_worktrees`
- Worktree workflow agents share personas via symlinks to the original workflow's agent directories — `existsSync()` follows symlinks, so workspace file validation passes without duplicating files
