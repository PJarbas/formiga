## Codebase Patterns
- Tests use `node:test` with `describe`/`it`, `node:assert/strict`
- DB tests create temp HOME directories, build minimal legacy schemas, then run migration via `getDb()` in a child process (`spawnSync` with `--input-type=module`)
- Migration uses `PRAGMA table_info` to detect existing columns before ALTER TABLE; for new tables uses `SELECT name FROM sqlite_master WHERE type='table'`
- DB module uses `node:sqlite` DatabaseSync, exports `getDb()`, `getDbPath()`, `nextRunNumber()`
- Existing migration pattern: `CREATE TABLE IF NOT EXISTS` for new tables, `PRAGMA table_info` + `ALTER TABLE ADD COLUMN` for new columns on existing tables
- Singleton tables use `CHECK (id = 1)` to enforce single-row constraint; `INSERT OR IGNORE` for idempotency
- Atomic counter updates use `UPDATE ... SET col = col + ? ... RETURNING col`
- INTEGER PRIMARY KEY in SQLite does NOT set `notnull=1` in PRAGMA table_info (it's an alias for ROWID)
- Dashboard tests use `createDashboardServer(0)` on random port with temp `HOME`/`TAMANDUA_DB_PATH`
- `getDb()` handles directory creation with `fs.mkdirSync(dirname, {recursive: true})`
- Legacy DBs without tamandua_stats are handled by `getSystemTokenSpend()` returning 0 (row null check + ?? 0 fallback)
- E2E tests: create temp HOME, write fake pi scripts, run `executePollingRound` via `runNodeScript` in child process, assert DB state via returned JSON
- `emitEvent()` writes to `<runId>.jsonl` event file; system events use runId="system"
- Dynamic imports in agent-scheduler (`await import("../db.js")`) for DB access from scheduler functions
- `incrementSystemTokenSpend` uses `RETURNING` to return new total for event emission
- Integration tests combine polling round execution + dashboard server startup + DB verification in single child process scripts
