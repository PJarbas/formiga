## Codebase Patterns
- Tests use `node:test` (built-in), not vitest or jest
- Test isolation: use `fs.mkdtempSync()` for temp HOME, `reserveDistinctRandomPorts()` for ports
- CLI invocations use `cleanChildEnv` from `tests/helpers/test-env.ts`
- E2E smoke tests use `path.join(process.cwd(), "dist", "cli", "cli.js")` for cliPath
- Workflow steps are advanced via `tamandua step claim` / `tamandua step complete` with canned outputs for smoke tests
- Run IDs are captured as 8-char prefixes from `tamandua workflow run` stdout, then resolved to full UUIDs via SQLite DB query
- `prepareGitRepo` creates isolated git repos from `e2e-tests/fixtures/sample-project/`
- E2e test shell scripts: set -euo pipefail, cd to repo root, run npm run build first, pipe output with tee to a LOG file, exit with PIPESTATUS
- E2e scripts use environment variables for log paths: TAMANDUA_E2E_TEST_LOG, TAMANDUA_SMOKE_E2E_LOG, TAMANDUA_REAL_E2E_LOG
- Daemon isolation: spawn `dist/server/daemon.js` directly with HOME + TAMANDUA_CONTROL_PORT in env, watch stdout for "control plane listening" to know it's ready
- `dashboard.log` is only created by daemonctl's `startDaemon()` wrapper (which opens the log fd for stdio), NOT by spawning daemon.js directly — direct spawning sends output to stdio pipes
- `os.homedir()` on Unix respects the HOME env var → setting HOME in child env is sufficient for daemon isolation
- Polling pattern: `spawnSync` CLI call in a loop with `await sleep(interval)`, parse stdout for `Status: <value>` line
- Structural e2e tests live in `tests/e2e-infrastructure.test.ts` and verify file/script structure WITHOUT running slow tests
- Real e2e test files use env vars for pi config (model, provider, apiKey) via `piAgentSettings()` helper
- After feature-dev-merge-worktree completes, the origin repo's main branch has a squash merge commit — verify by checking the origin repo files/git log, not the worktree
- Sequential real e2e tests use `before`/`after` hooks for shared state (temp HOME, repo), with per-test daemon lifecycle (start→work→stop→restart)
- Documentation consistency (AGENTS.md e2e guidance) is verified structurally via `tests/e2e-infrastructure.test.ts` — test that the markdown file contains specific directives, script references, and cost/duration info without parsing rendered output
- Worktree directories are created under ~/.tamandua/worktrees/ (resolved via resolveWorktreeRoot() which reads os.homedir() or TAMANDUA_WORKTREE_ROOT env var); with isolated HOME, they're under the temp HOME and cleaned up by cleanupTempHome
