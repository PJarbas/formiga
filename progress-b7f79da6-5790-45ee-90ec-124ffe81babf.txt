## Codebase Patterns
- CLI entry point is `src/cli/cli.ts` with `main()` dispatch
- CLI tests use `spawnSync` with the `bin/tamandua` shell wrapper
- Tests always use isolated temp HOME and TAMANDUA_STATE_DIR
- `printUsage()` writes to stdout via `process.stdout.write`
- The `shouldSkipUpdateWarning()` function suppresses version warnings for certain commands
- Build: `npm run build` (tsc), then tests: `npm test`
- `hasHelpFlag(args)` checks for `--help` or `-h` anywhere in args
- `printHelp(text)` writes text to stdout and exits 0
- `printHelpSubcommand(subcommands)` renders subcommand listing
- Help texts are standalone functions returning multiline strings
- Command-specific help uses `printHelp()` (which exits 0); unreachable fallback is `printHelp(getUsageText())`
- Some commands have CLI aliases (version: `--version`, `-v`) that must be handled in help dispatch
- Regex in `assert.match()` uses `.` which does NOT match newlines; use individual assertions for multi-line terms
- Step subcommand help dispatch uses `if (group === "step")` with per-action `if` checks (not if/else if) so fallthrough to global usage works for unknown actions
- Service command groups (mcp, dashboard, control-plane) follow the same subcommand-dispatch pattern as step: sequential `if` blocks for start/stop/status, with a group-level fallback `printHelp()`
- Default ports: MCP=3338, dashboard=3334, control-plane=3339
- Workflow subcommand dispatch uses the same pattern: sequential `if` blocks for each action, with group-level fallback `printHelp(getWorkflowGroupHelp())`
