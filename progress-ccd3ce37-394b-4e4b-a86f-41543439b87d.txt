## Codebase Patterns
- All 18 bundled workflows live under workflows/ directory
- Workflow families: Feature Development (feature-dev*), Bug Fix (bug-fix*), Security Audit (security-audit*), Quick Tasks (do-now, just-do-it, do-review-do-verify)
- Each feature-dev variant: 5-6 agents (planner, setup, developer, verifier, tester, +merger for merge variants, +reviewer for GitHub PR)
- Each bug-fix variant: 5-6 agents (triager, investigator, setup, fixer, verifier, +merger for merge)
- Each security-audit variant: 6-7 agents (scanner, prioritizer, setup, fixer, verifier, tester, +merger for merge)
- Worktree variants use `run.workspace: worktree` and run in detached git worktrees
- README.md workflow section was outdated: listed only 7 of 18 workflows, wrong agent counts, wrong pipeline diagrams
- Agent counts from workflow.yml `agents` array, pipeline steps from `steps` array
- Tests use `loadWorkflowSpec()` to parse workflow.yml into spec objects with agents/steps arrays
- Existing tests in `tests/readme-workflow-catalog.test.ts` verify README contains workflow IDs but not pipeline accuracy
- MCP tool names are defined as module-private constants in src/server/mcp-server.ts (not exported); tests hardcode the 9 expected tool names
- `npm test` runs the full parallel suite; individual test files can be run with `node --test tests/<file>.test.ts`
- Harness flags (--pi-as-harness / --hermes-as-harness) are parsed in src/cli/workflow-run-args.ts, mutually exclusive
- TAMANDUA_HERMES_BINARY env var and fallback PATH search for hermes binary in src/installer/agent-scheduler.ts findHermesBinary()
- Harness binary validation runs at scheduling time in src/installer/run-harness.ts
