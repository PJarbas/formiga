## Codebase Patterns

- Tests import from `../../dist/` (compiled JavaScript), not directly from source TypeScript
- Tests use `fs.mkdtempSync` for temp HOME isolation with `before`/`after` hooks
- The `getDb()` function in `db.ts` is synchronous (uses `DatabaseSync`), no async needed
- Dynamic `import()` works in ESM test files; `require()` does not
- Agent-scheduler uses `await import("../db.js")` for lazy module loading to avoid circular deps
- `run-harness.ts` can import from `agent-scheduler.ts` without circular deps (neither imports the other)
- Context is stored as JSON TEXT in `runs.context` column — no schema migration needed for new fields
- Parsing boolean flags in parseWorkflowRunArgs: use `if (token === "--flag") { variable = true; continue; }` pattern
- The tamandua CLI entry point is a shell wrapper at `bin/tamandua` — invoke with `sh`, not `node`
- findPiBinary() serves as the template for findHermesBinary(): env var check → PATH search → descriptive error
- runPi() serves as the template for runHermes(): spawn with detached:true, SIGTERM/SIGKILL timeout, lifecycle logging
- runHermes() differs from runPi() in: no JSON parsing (hermes outputs plain text), session_id filtering via /^session_id:\s*\S+/ regex
- executePollingRound tests need: DB with runs table (task/created_at/updated_at NOT NULL), mock pi/hermes binaries, workingDirectoryForHarness on CronJobInfo
- findHermesBinary() is synchronous (uses fs.accessSync); tests must use assert.throws not assert.rejects
- createAgentCronJob() reads harness_type from DB runs.context JSON to populate CronJobInfo.harnessType; defaults to "pi"
- _scheduledRunIds() is exported for test assertions on scheduler state
